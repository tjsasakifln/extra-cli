**Refs:** `docs/architecture/adr/ADR-007-cloud-hosting-strategy.md`, `docs/ops/cloud-deployment-plan.md`, `docs/guides/hetzner-supabase-plan.md` (histórico)

## Decisão

| Dimensão | Escolha | Justificativa |
|----------|---------|---------------|
| **Provedor preferencial** | **Netcup** Root Server (~RS 4000 class) | CPU dedicada, 32 GB RAM, ~1 TB NVMe, melhor custo para volume 4M+ contratos (ADR-007) |
| **Provedor alternativo** | **Hetzner Cloud** (CX/CCX ou dedicated) | API/`hcloud`/OpenTofu maduros; fallback se Netcup falhar em compra/SLA/PNCP |
| **Região preferencial** | Costa leste EUA **ou** Brasil (se disponível no provedor) | Latência crawlers vs APIs BR; **condicionada a teste PNCP** (ADR-007 §1) |
| **Região fallback** | Europa (Hetzner Nuremberg/Helsinki) | Só se teste PNCP for OK a partir da UE |
| **SO** | Ubuntu 24.04 LTS | Alinhado a `deploy/provision-vps.sh` |
| **DB** | PostgreSQL 16 bare-metal na VPS | Sem Supabase obrigatório no estágio inicial (doc histórico desatualizado) |
| **App** | Python 3.12 + systemd timers | Inventário em `docs/audits/vps-production-readiness-2026-07.md` |
| **Dimensão inicial** | **32 GB RAM**, CPU dedicada ou alto share, **≥500 GB–1 TB NVMe** | Referência ADR-007; monitorar disco/bloat |
| **Backup remoto** | Object storage / Storage Box do mesmo provedor ou S3-compatible | V6.2 lista credenciais |
| **Não escolher agora** | Kubernetes, Kafka, Redis cluster, multi-região | YAGNI até métricas |

---

## Dimensionamento operacional (mínimo aceitável)

| Recurso | Mínimo | Alvo inicial |
|---------|--------|--------------|
| vCPU | 4 | 8 dedicados / equivalente |
| RAM | 16 GB | **32 GB** |
| Disco | 250 GB NVMe | **1 TB** NVMe |
| Swap | 4 GB | 8 GB |
| Rede | 1 Gbps | IPv4 público + firewall |

Workload: PostgreSQL + ~20 systemd crawl/timers + relatórios + health checks.

---

## Critérios de go/no-go antes de pagar (liga V6.2)

1. Teste PNCP real (`monitor.py --source pncp`) a partir do DC candidato — registrar 2xx/4xx/latência.
2. Conta provedor criada + meio de pagamento (ação humana Tiago).
3. SSH key-only + firewall script de `deploy/` validado em dry-run local.
4. Plano de backup: dump diário + retenção 7d + restore drill (já provado localmente).
